## 1. 시스템 전체 구조

### 🏭 "이 시스템은 거대한 물류 관제 센터입니다"

이해를 돕기 위해, 우리 시스템을 **수백 대의 배달 오토바이를 운용하는 대형 물류 회사**에 비유해 보겠습니다.

---

### 📖 한 편의 이야기로 이해하는 데이터 흐름

> **아침 9시, 배달 기사 김철수 씨가 오토바이에 시동을 겁니다.**

---

**① 🏍️ 오토바이 (데이터 발생)**

오토바이에는 작은 전자 장치 하나가 붙어 있습니다. 이 장치는 마치 오토바이의 **비서**처럼, 1초에 한 번씩 이런 정보를 자동으로 기록합니다.

- "지금 위치는 서울시 강남구 테헤란로 123번지야"
- "현재 속도는 시속 45km야"
- "배터리는 72% 남았어"
- "엔진 온도는 정상이야"

이 비서는 쉬지 않고 이 정보를 **본사 관제 센터**로 전송합니다.

---

**② 📡 통신망 (데이터 전달)**

오토바이가 보낸 정보는 어떻게 관제 센터까지 도달할까요?

마치 **무전기**처럼, 오토바이의 비서 장치는 인터넷을 통해 본사 서버에 계속 신호를 보냅니다. "삐삐삐— 저 여기 있어요, 이렇게 달리고 있어요!" 하고요.

---

**③ 🏢 본사 서버 (데이터 처리)**

본사 서버는 전국 수백 대의 오토바이에서 동시에 날아오는 신호를 받아서 세 가지 일을 합니다.

1. **기록**: "어, 오토바이 52호차가 지금 강남에 있군. 속도 45km. 배터리 72%. 기록!"
2. **판단**: "잠깐, 오토바이 17호차 속도가 갑자기 시속 90km? 제한속도 60km를 초과했네. 경고 발령!"
3. **알림**: 관제사의 화면과 기사의 스마트폰에 동시에 경고 메시지를 전송.

---

**④ 📊 관제 대시보드 (사람이 보는 화면)**

본사의 관제사는 커다란 모니터 화면 하나로 전국 모든 오토바이의 상태를 실시간으로 봅니다.

- 지도 위에 오토바이 아이콘들이 살아서 움직이고 있어요.
- 빨간 아이콘 = 지금 문제가 생긴 차량
- 초록 아이콘 = 정상 운행 중인 차량
- 노란 아이콘 = 주의가 필요한 차량 (배터리 부족 등)

마치 항공 관제탑에서 비행기들을 모니터링하는 것처럼요. ✈️

---

**⑤ 📱 기사용 모바일 앱 (현장 연결)**

기사 김철수 씨의 스마트폰에도 같은 시스템이 연결되어 있습니다. 배터리가 20% 이하로 떨어지면, 본사 서버가 자동으로 김 씨의 폰에 메시지를 보내요.

> *"앞으로 3km 이내에 충전소가 2곳 있습니다. 왼쪽 버튼을 누르면 길 안내를 시작합니다."*

김 씨가 버튼 하나를 누르면, 내비게이션 앱이 자동으로 열리며 최적 경로가 안내됩니다.

---

## 2. 주요 기술 요소 해설

### 🔍 IoT 센서 — 오토바이의 오감

> **기술 용어**: GPS 센서, OBD 포트, BMS(배터리 관리 시스템)

---

오토바이에는 사람의 **오감(눈, 코, 귀, 입, 피부)**에 해당하는 여러 센서가 달려 있습니다.

| 사람의 감각 | 오토바이의 센서 | 하는 일 |
|---|---|---|
| 👁️ **눈** | GPS 센서 | "나는 지금 서울 강남구 테헤란로에 있어" — 위치를 파악합니다 |
| 🖐️ **피부** | OBD 포트 | "엔진 온도가 높아. 브레이크 패드가 닳았어" — 차량 상태를 느낍니다 |
| 🫀 **심장박동** | BMS (배터리) | "배터리 잔량 45%, 충전까지 약 22km 남았어" — 에너지 상태를 체크합니다 |
| 🦵 **근육 감각** | 속도·가속도 센서 | "지금 시속 67km, 급가속 감지!" — 움직임을 인식합니다 |
| 🌡️ **체온계** | 온도·습도 센서 | "기온 34도, 폭염 주의 구간" — 환경 조건을 측정합니다 |

이 센서들이 수집한 정보는 **1초에 1번씩** 자동으로 서버로 전송됩니다. 사람이 개입하지 않아도, 오토바이 스스로 "나 괜찮아요~"를 계속 보고하는 것이죠.

---

### 📡 MQTT & WebSocket — 무전기와 생방송 중계

> **기술 용어**: MQTT 프로토콜, WebSocket 실시간 통신

---

#### 🎙️ MQTT — "끊기지 않는 무전기"

경찰이나 소방관이 사용하는 **무전기**를 생각해 보세요.

일반 전화기는 두 사람이 1:1로 통화해야 하지만, 무전기는 **한 사람이 말하면 채널에 연결된 모든 사람이 동시에 듣습니다**. 게다가 연결을 항상 유지하고 있기 때문에 즉시 통화가 가능해요.

우리 오토바이의 IoT 장치는 정확히 이런 방식으로 작동합니다.

- 오토바이 장치가 1초마다 짧은 신호를 보냅니다: **"위치: 37.5°N, 127.0°E / 속도: 42km/h / 배터리: 68%"**
- 이 신호는 일반 인터넷보다 훨씬 **가볍고 빠른 무전기 방식(MQTT)**으로 전송됩니다.
- 데이터 양이 극히 적어서 (문자 메시지 1개보다 작아요) 이동 중에도 끊기지 않습니다.

> 💡 **비유**: 배달 기사가 매 1분마다 본사에 전화해서 "나 지금 여기야"를 보고하는 대신, 허리춤에 찬 GPS 단말기가 자동으로 계속 신호를 보내는 것과 같아요. 기사는 운전에만 집중하면 됩니다.

---

#### 📺 WebSocket — "끊기지 않는 생방송 채널"

관제사가 모니터를 보는 화면은 어떻게 오토바이의 위치가 **실시간으로** 움직이는 걸까요?

일반 웹사이트는 이런 방식입니다.
> "사용자가 새로고침 버튼을 누른다 → 서버에 요청 → 서버가 응답 → 화면 업데이트"

하지만 지도에서 오토바이가 실시간으로 움직이려면, 새로고침 없이도 **화면이 알아서 바뀌어야** 합니다.

이것을 가능하게 하는 것이 **WebSocket**인데, 이것은 마치 **TV 생방송 채널**과 같습니다.

- TV를 켜면 방송국과 연결이 됩니다.
- 내가 리모컨을 누르지 않아도, 화면은 계속 바뀝니다.
- 방송국에서 뭔가 새로운 일이 생기면, 즉시 내 TV에 표시됩니다.

우리 관제 대시보드도 마찬가지예요. 관제사가 화면을 열면, 서버와 생방송 채널이 연결되고, 오토바이가 움직일 때마다 서버가 **알아서 화면을 업데이트**해 줍니다.

> 💡 **결과적으로**: 오토바이 → (무전기 방식으로) → 서버 → (생방송 채널로) → 관제사 화면. 이 과정이 1초 이내에 완성됩니다.

---

### 🍽️ REST API — 식당의 웨이터

> **기술 용어**: REST API, HTTP 요청/응답

---

관제사가 화면에서 "이번 달 운행 기록 보여줘"라는 버튼을 눌렀을 때, 어떤 일이 벌어질까요?

이것을 **식당**에 비유하면 정말 쉽습니다.

```
고객 (관제사 화면)
  │
  │ "강남구 52번 차량의 이번 달 운행 기록 주세요"
  │         (메뉴판에서 주문)
  ▼
웨이터 (REST API)
  │
  │ 주문서를 들고 주방으로 이동
  ▼
주방 (서버 & 데이터베이스)
  │
  │ 데이터를 꺼내 예쁘게 담아서 반환
  ▼
웨이터 (REST API)
  │
  │ 요리를 고객에게 전달
  ▼
고객 화면
  화면에 운행 기록표가 나타납니다 ✅
```

**핵심 특징**: 웨이터는 고객이 주문할 때만 움직입니다. 주문이 없으면 가만히 서 있어요.

이것이 실시간 무전기(MQTT)와 다른 점입니다.

| 구분 | 실시간 통신 (무전기) | API 요청 (웨이터) |
|---|---|---|
| 언제 작동? | 항상 연결, 데이터가 오면 즉시 전달 | 사용자가 요청할 때만 |
| 용도 | 오토바이 위치 실시간 표시 | 운행 기록 조회, 차량 등록, 알림 확인 등 |
| 비유 | 생방송 TV | 배달 주문 |

### 🍳 AWS 서버리스 — 공유 주방

> **기술 용어**: AWS Chalice, 서버리스(Serverless) 컴퓨팅

---

#### 일반 서버 vs 서버리스의 차이

**기존 방식 — "24시간 식당 임대"**

전통적인 IT 시스템은 **식당 건물 전체를 임대**하는 것과 같습니다.
- 손님이 1명이든 1,000명이든, 임대료는 똑같이 냅니다.
- 새벽 3시에 손님이 없어도 전기, 인건비, 임대료가 나갑니다.
- 갑자기 손님이 100배 늘어나면, 새 건물을 추가로 임대해야 합니다.

**우리 방식 — "공유 주방 (Cloud Kitchen)"**

우리가 선택한 방식은 마치 **배달 전문 공유 주방**과 같습니다.

공유 주방은 이렇게 작동해요.

- 요리가 필요할 때만 주방을 빌립니다. 요리를 안 하면 돈을 내지 않아요.
- 주문이 갑자기 10배 늘어도, 공유 주방 운영사가 알아서 공간을 더 열어줍니다.
- 설거지, 청소, 시설 유지는 공유 주방 운영사(AWS)가 모두 합니다.

```
🍕 주문 들어옴 (사용자 요청 발생)
      ↓
🔥 공유 주방 가동 (서버 실행 — 0.1초 만에)
      ↓
🍴 요리 완성 (데이터 처리 완료)
      ↓
💤 주방 원상복구 (서버 종료 — 비용 중단)

주문이 없는 동안 → 비용 0원
```

> 💡 **비즈니스 핵심**: 일반 서버는 오토바이가 1대든 1,000대든 매달 고정 비용이 발생합니다. 하지만 공유 주방 방식은 **오토바이 수가 늘어날수록 비용도 늘지만, 줄어들면 비용도 줄어듭니다.** 사업 초기 리스크를 대폭 낮출 수 있어요.

### 📚 데이터베이스 — 회계 장부와 CCTV 녹화

> **기술 용어**: SQLModel (관계형 DB), TimescaleDB (시계열 DB)

---

우리 시스템의 데이터 저장소는 **두 종류의 기록 시스템**으로 나뉩니다.

#### 📒 관계형 데이터베이스 — "꼼꼼한 회계 장부"

회사의 **회계 담당자**가 쓰는 장부를 생각해 보세요.

- 직원 명부: 홍길동, 주민번호 123456, 직책 배달기사, 입사일 2024-01-15
- 차량 대장: 오토바이 52호, 차종 혼다CB, 취득일 2023-06-01, 담당기사 홍길동
- 각 항목들이 서로 연결되어 있어요. "홍길동이 52호차를 담당한다"는 관계가 기록됩니다.

이렇게 **체계적으로 분류되고 서로 연결된 정보**를 저장하는 것이 관계형 데이터베이스입니다.

> "52호차 담당 기사가 누구야?" → 장부를 보면 바로 홍길동이라고 나옵니다.

#### 📹 시계열 데이터베이스 — "CCTV 녹화 테이프"

오토바이가 1초마다 보내는 위치·속도·배터리 정보는 **CCTV 영상**과 같습니다.

- 1초마다 새로운 데이터가 쌓입니다. 차량 1대 = 하루 86,400개의 기록.
- 100대라면 하루에 **864만 개**의 데이터가 쌓여요.
- 이 데이터는 **시간 순서가 핵심**입니다. "오전 9시 43분 12초에 강남 지점에 있었다"는 기록이죠.

일반 장부로는 이런 방대한 기록을 빠르게 찾기가 어려워요. 그래서 **CCTV 전용 저장 장치**처럼, 시간 기반으로 최적화된 특수 데이터베이스를 씁니다.

```
일반 DB (회계 장부)          시계열 DB (CCTV 녹화)
─────────────────────        ──────────────────────────────
차량 마스터 정보             09:00:01 → 위치A, 속도42, 배터리72%
사용자 계정 정보             09:00:02 → 위치B, 속도43, 배터리72%
충전소 위치 정보             09:00:03 → 위치C, 속도41, 배터리71%
운행 요약 기록               09:00:04 → 위치D, 속도45, 배터리71%
                             ...
                             (1초마다 무한 반복)
```

> 💡 **보존 정책**: CCTV 영상도 일정 기간 후 덮어쓰듯, 우리 시스템도 3개월이 지난 상세 기록은 자동으로 정리됩니다. 단, 중요한 사고 기록이나 요약 데이터는 영구 보존됩니다.

### 🧠 AI 분석 엔진 — 숙련된 안전 관리 매니저

> **기술 용어**: AI 예측 모델, 이상 감지 알고리즘

---

#### 신입 관리자 vs 10년 경력 베테랑의 차이

**신입 관리자**는 이렇게 생각해요.
> "규정상 제한속도는 60km/h니까, 61km/h만 되면 경고를 울리자."

**10년 경력 베테랑**은 이렇게 생각해요.
> "이 기사는 평소에 항상 45~50km/h로 달리는데, 갑자기 70km/h가 됐네? 뭔가 이상한 상황인 것 같은데. 그리고 배터리가 평소보다 30% 더 빨리 줄고 있어. 아마 언덕 구간을 달리거나 배터리가 손상됐을 가능성이 높아."

우리 AI 엔진은 **경험 많은 안전 관리 매니저**처럼 행동합니다.

---

#### AI가 하는 3가지 핵심 역할

**1. 📊 패턴 학습 — "이 기사는 원래 이렇게 달려"**

AI는 각 기사와 차량의 **평소 패턴**을 기억합니다.
- 홍길동 씨는 평균 속도 43km/h, 배터리 소모율 하루 60%
- 갑자기 오늘 배터리가 하루에 85% 닳는다면? → "이상하다! 점검 필요"

**2. ⚠️ 조기 경보 — "배터리 방전 1시간 전에 미리 알려줘"**

단순히 "배터리 20% 이하"일 때 경고하는 게 아니라, AI는 **소모 추세를 보고 미리 계산**합니다.

```
현재 배터리: 45%
현재 소모 속도: 분당 1.2%
예상 방전 시점: 약 37분 후

→ "지금 출발하면 배달 완료 전에 방전됩니다. 
   출발 전 충전을 권장합니다." 경고 발령 ⚡
```

**3. 🔮 예측 유지보수 — "이 차, 다음 주에 고장 날 것 같아"**

CCTV 기록(시계열 데이터)을 분석하여, 엔진 진동 패턴이나 브레이크 반응 시간 변화를 감지합니다.
- 사람이 느끼기 전에 **미세한 변화**를 포착
- "52호차, 브레이크 패드 교체 시기가 2주 이내로 예상됩니다" 알림

> 💡 **한 줄 요약**: AI 엔진은 "현재 상태가 이상하냐 아니냐"를 판단하는 게 아니라, "**앞으로 어떤 일이 생길 것 같은지**"를 미리 알려주는 것이 핵심 가치입니다.

## 3. 사용자 화면 해설

### 🎨 "매장의 인테리어와 계기판"

프론트엔드(개발 용어)를 쉽게 말하면, **"고객이 실제로 보고 사용하는 모든 화면"**입니다. 마치 자동차의 계기판처럼요. 계기판은 엔진이 어떻게 작동하는지 몰라도, 숫자와 색상만 보면 상태를 알 수 있게 설계되어 있죠. 🚗

---

### 🚦 색상의 의미 — 직관적인 신호등 시스템

우리 화면의 모든 색상은 **신호등**과 동일한 논리를 따릅니다. 별도로 배우지 않아도 직관적으로 이해할 수 있어요.

| 색상 | 의미 | 실제 화면에서의 쓰임새 |
|---|---|---|
| 🟢 **초록** | 정상 / 안전 / 완료 | 정상 운행 중인 차량, 충전 완료, 목표 달성 |

| 🔴 **빨강** | 위험 / 즉시 조치 | 과속 감지, 사고 감지, 배터리 10% 이하 |
| 🔵 **파랑** | 정보 / 일반 알림 | 운행 시작/종료, 배차 완료, 일반 공지 |
| ⚫ **회색** | 비활성 / 오프라인 | 전원 꺼진 차량, 통신 두절 |

## 4. 비즈니스 가치

### 💰 왜 이 기술이 돈이 되고, 비용을 줄이고, 사고를 예방하나요?

---

### 관점 1 💵 "왜 돈이 되는가" — 데이터가 새로운 수익을 만든다

**지금까지의 오토바이 사업**: 오토바이를 사서, 기사에게 줘서, 배달시키고, 수수료를 받는다.

**FMS를 도입한 후**: 그 이상의 것을 팔 수 있게 됩니다.

---

🏪 **1. 운행 데이터 기반의 보험 상품**

> "이 기사는 6개월간 과속이 단 2회, 급정거 없음. 안전 운전자입니다."

이 데이터를 보험사에 제공하면, 기사별로 **맞춤형 보험료** 책정이 가능해집니다. 안전 기사는 보험료가 낮아지고, 이는 기사 모집의 경쟁력이 됩니다.

---

📊 **2. 운영 효율성 보고서 판매**

> "A 지역은 오후 2시~4시에 배달 수요가 집중되지만, 우리 차량은 이 시간에 평균 23% 비어 있습니다."

이런 분석 리포트를 기업 고객(B2B)에게 제공하는 **운영 컨설팅 서비스**로 발전시킬 수 있습니다.

---

🔋 **3. 충전 네트워크 제휴 수익**

AI가 "이 차량은 3시간 후에 충전이 필요할 것"을 미리 예측하면, 충전소를 미리 예약하고 최적 경로를 안내할 수 있습니다. 충전소 운영사와 **예약 수수료 수익** 모델을 만들 수 있어요.

---

### 관점 2 📉 "어떻게 비용을 줄이는가" — 3가지 숨은 비용의 제거

**🔧 비용 절감 1: 예방 정비 vs. 고장 후 수리**

| 구분 | 고장 후 수리 (기존) | 예방 정비 (FMS 도입) |
|---|---|---|
| 수리비 | 평균 50만원 | 평균 8만원 (부품 교체) |
| 차량 운행 중단 | 3~5일 | 반나절 |
| 기사 수입 손실 | 3~5일치 | 없음 |
| **총 손실** | **약 100만원** | **약 8만원** |

AI가 "다음 주에 브레이크 패드 교체 필요" 알림을 주면, **미리 예약 정비**를 할 수 있습니다. 차량 한 대당 연간 수백만 원의 절감이 가능합니다.

---

**⛽ 비용 절감 2: 배터리 관리 최적화**

전기 오토바이의 가장 큰 비용은 **배터리 수명**입니다. 배터리 하나에 100~200만원이죠.

- **급속 충전 과다 사용** → 배터리 수명 30% 단축
- FMS는 각 기사의 충전 패턴을 모니터링하고, "급속 충전을 너무 자주 사용하고 있습니다. 완속 충전 비율을 높이면 배터리 수명이 8개월 더 늘어납니다"를 알려줍니다.

> 차량 100대 기준, 배터리 교체 주기를 6개월만 늘려도 **연간 1억 원 이상** 절감.

---

**👷 비용 절감 3: 행정 자동화**

기존에는 사람이 직접 해야 했던 일들이 자동화됩니다.

| 기존 업무 | FMS 도입 후 |
|---|---|
| 기사가 매일 운행일지 수기 작성 | 자동 생성, 서명만 하면 끝 |
| 관리자가 엑셀로 운행 기록 집계 | 실시간 대시보드로 즉시 조회 |
| 월말 연료비/충전비 정산 | 자동 집계 및 보고서 생성 |
| 차량 위치 확인을 위한 기사 전화 | 지도에서 실시간 확인 |

중소 물류사 기준, 이 자동화만으로도 **행정 인력 1~2명 분량**의 업무가 줄어듭니다.

---

### 관점 3 🛡️ "어떻게 사고를 예방하는가" — 사고는 막는 것이 최선

배달 오토바이 사고의 **3대 원인**과 FMS의 대응 방식입니다.

---

**🚨 사고 원인 1: 과속 및 난폭 운전**

- **기존**: 사고가 난 후에야 블랙박스 영상을 확인합니다.
- **FMS**: 과속이 감지되는 **그 순간** 기사 폰과 관제 화면에 동시 경고. 반복 적발 시 관리자가 즉시 연락 가능.
- **효과**: 과속 감지 경고 시스템 도입 후 평균 과속 빈도 **60~70% 감소** (유사 솔루션 도입 기업 사례 기준)

---

**🔋 사고 원인 2: 배터리 방전으로 인한 도로 위 고립**

- **기존**: 배터리가 완전히 방전되어 기사가 도로 한복판에 멈춰서, 후방 추돌 위험.
- **FMS**: 방전 **1시간 전**에 자동 알림 + 주변 충전소 안내. 기사가 미리 대응 가능.
- **효과**: 도로 위 방전 사고 **원천 차단**

---

**🌧️ 사고 원인 3: 기상·도로 상황 무시한 무리한 운행**

- **기존**: 폭우/폭설 상황에서도 배달 압박으로 무리하게 운행.
- **FMS**: 기상 데이터 연동으로 위험 구간에 진입 시 자동 속도 제한 권고. 관리자가 전체 차량에 "현재 날씨로 인해 서행 운행 중"으로 일괄 공지 가능.
- **효과**: 악천후 사고율 감소, 배달 파트너사와의 계약 시 안전 관리 체계 증빙 자료로 활용

---

### 📌 결론 — 한 문장으로 요약

> 🏍️ **지능형 오토바이 FMS는 단순한 "위치 추적 시스템"이 아닙니다.**  
> 오토바이 한 대가 달리는 모든 순간의 데이터를 수집하고, AI가 분석하여,  
> **사고를 미리 막고, 비용을 자동으로 줄이며, 운영 효율을 데이터로 증명하는** 종합 플랫폼입니다.

## 부록. 자주 묻는 질문 (FAQ)

**Q. 오토바이 기사가 위치 추적을 거부하면 어떻게 하나요?**  
A. 앱 가입 시 위치 정보 제공에 동의한 기사만 배차를 받을 수 있는 구조입니다. 수집된 위치 데이터는 업무 목적 외 사용이 금지되며, 이를 개인정보 처리방침에 명시합니다. 🔒

**Q. 인터넷이 끊기면 데이터가 사라지나요?**  
A. 아닙니다. 오토바이 장치에 내부 저장 공간이 있어서, 통신이 일시 두절되어도 데이터를 임시 저장합니다. 인터넷이 다시 연결되면 그동안 쌓인 데이터를 자동으로 전송합니다. 📦

**Q. 데이터가 해킹당할 위험은 없나요?**  
A. 모든 데이터는 암호화되어 전송 및 저장됩니다. 또한 AWS(아마존 웹 서비스)의 보안 인프라를 사용하는데, 이는 전 세계 은행과 정부기관도 사용하는 수준입니다. 🛡️

**Q. 시스템 도입 비용은 어느 정도인가요?**  
A. 오토바이당 장착하는 IoT 장치 비용 + 월별 클라우드 사용료 구조입니다. 차량 수가 늘면 규모의 경제로 단가가 낮아집니다. 정확한 견적은 운용 규모에 따라 별도 협의합니다. 💬
